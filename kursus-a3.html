Avanceret system-arkitektur

Multi-agent systems, model routing, distribuerede AI-pipelines og det svære valg: hvornår rigtige agenter slår en simpel workflow.

⏱ 2 – 2,5 timer
📋 4 moduler
🎯 Afsluttes med mini-quiz
Del A3
Når én model ikke er nok

På B-niveau lærte du at bygge én agent (B4). På A-niveau bygger du systemer af agenter og modeller der samarbejder. Det er forskellen mellem en intern automation og et rigtigt AI-produkt der skalerer til tusind brugere.

1
Multi-agent systemer: hvornår 5 agenter slår 1
Den dyreste arkitektur-fejl: at bygge multi-agent når en workflow ville have virket
+

I 2024 troede alle multi-agent var fremtiden. I 2026 ved professionelle teams at de fleste multi-agent-projekter fejler — ikke fordi konceptet er forkert, men fordi det blev brugt til problemer en simpel workflow (B4 mønster 1-2) løste bedre.

Beslutnings-rammen

De tre nøglespørgsmål
  1. Er antallet af steps kendt på forhånd? → workflow (B4)
  2. Kan opgaven nedbrydes i klart adskilte roller med forskellig ekspertise? → multi-agent kan vinde
  3. Er agenterne afhængige af hinandens output i mere end 1-2 ping-pongs? → STOP. Bliv ved workflow.
Mønster 1

Supervisor + worker

Én "supervisor"-agent koordinerer; flere "worker"-agenter har hver deres tools og kompetencer. Supervisor ruter beskeder, sammenlægger resultater. Det er det mest robuste multi-agent mønster.

Eksempel: research-agent
supervisor: "Find seneste tal for elbilsalg i Skandinavien." ├─ ruter til: web_search_agent │ → "elbilsalg Norge 2026" → returnerer artikler ├─ ruter til: data_extraction_agent │ → udtrækker tal fra artikler → returnerer JSON └─ ruter til: synthesis_agent → samler tabeller, formulerer svar
Mønster 2

Debate / multi-perspective

Flere agenter med forskellige roller diskuterer samme spørgsmål. Klassisk: "Optimist" + "Skeptiker" + "Dommer". Bruges til strategi, kode-review, beslutningsstøtte.

⚠ Latente problemer

Multi-agent systemer eksploderer i token-forbrug. 3 agenter × 5 turns × 2k tokens = 30k tokens per request. Det er ikke en hobby-arkitektur. Sæt hårde grænser (B5).

Anthropic's egen lære (offentliggjort 2024-2025)

Anthropic byggede flere multi-agent forskningsprojekter og skrev åbent om det: "De fleste agentic-systemer er bedre tjent med en simpel ReAct-loop og god prompt-engineering end med multi-agent koreografi." Brug multi-agent når problemets natur kræver det — ikke fordi det "lyder cool".

🔥 PRØV DET NU — Vurdér tre scenarier
  1. "Skriv en blog-post med researcher der finder fakta og editor der finpudser sproget."
  2. "Booket en flybillet inkl. hotel og leje af bil baseret på en ferieplan."
  3. "Diskutér om vi bør lancere produktet i Q3 eller Q4 — fra finans-, marketing- og produkt-perspektiv."

For hver: workflow (B4 mønster 1-2), single agent (B4 mønster 3), eller multi-agent (A3 mønster 1-2)? Hvorfor?

Hint: 1) workflow chaining. 2) single agent med tools. 3) multi-agent debate.

2
Model routing og cascading i produktion
Brug ikke en Ferrari til at hente brød
+

På B5 mødte du model cascading kort: send det nemme til billige modeller, det svære til dyre. På A-niveau gør vi det systematisk med målbare gevinster.

De tre routing-strategier

1. Regel-baseret routing

If-then ud fra simple features (input-længde, sprog, intent-klasse). Hurtigt, billigt, forudsigeligt. Dur til 70% af cases.

2. Klassifikator-routing

En lille model (eller en LLM med en kort prompt) klassificerer requesten først: "let / svær". Derefter ruter du. Dette er hvor de fleste seriøse pipelines lander.

3. Cascading med fallback

Send altid til billig model først. Lad den selv-rapportere konfidens. Hvis konfidens er lav — eskaler til dyrere model. Bedre end klassifikator når svære cases er sjældne.

Et konkret cascading-design

Pseudocode
def route(query): answer, confidence = haiku.complete( query, prompt="Svar. Slut med 'KONFIDENS: HØJ/MIDDEL/LAV'." ) if confidence == "HØJ": return answer # 80% af cases — billigt if confidence == "MIDDEL": return sonnet.complete(query) # 18% — middelpris return opus.complete(query) # 2% — top-kvalitet
Effekt på pris og latens

Typisk reduktion: 60-80% pris, 40-60% latens. Kvaliteten på de svære queries er stadig "Opus-niveau" — fordi de svære blev sendt til Opus. Det her er hvad seriøse AI-features ser ud i 2026.

Den falske besparelse: latens-skat

⚠ Pas på

Cascading kan blive langsommere end at gå direkte til den dyre model — fordi når 20% eskalerer, har de allerede ventet på det første kald. Hvis brugeren venter live, så lav den første kald så hurtigt som muligt (Haiku, kort prompt) og brug streaming på fallback.

🔥 PRØV DET NU — Plan en cascade
  1. Tag et AI-projekt du kender (eget eller B-eksamen).
  2. Estimér: hvor mange % er "lette" queries? Klassificering, simpel parafrasering, korte svar.
  3. Hvilken model ville være "god nok" til 80% af dem?
  4. Beregn ny pris: 80% × billig + 20% × dyr — sammenlign med 100% × dyr.

Typisk besparelse på reelle projekter: 5-50× lavere månedlig regning.

3
Distributed AI: kø-systemer, async jobs og batch
Når brugeren ikke kan vente, men opgaven tager 2 minutter
+

Mange AI-opgaver tager længere end et HTTP-request burde leve. Generering af en lang rapport, processering af 50 PDF'er, fine-tuning. Synchronous design lokker dig i fælder. På A-niveau designer du async-først.

De tre arkitektur-mønstre

1. Job queue (typiske opgaver)

Bruger trykker "Start". Backend lægger jobbet på en kø (SQS, Redis, Cloud Tasks). En worker plukker det. Frontend poller eller modtager websocket-notifikation når det er færdigt.

2. Streaming (delvis output med det samme)

Bruger ser tegn dukke op fra første sekund. Følt latens er ~0,5s i stedet for 8s. Første-token-time er det metric der betyder noget — ikke total tid.

3. Batch-API'er (offline jobs)

OpenAI's og Anthropic's batch-endpoints accepterer 50.000 requests og leverer indenfor 24 timer — til halv pris. Perfekt til ad-hoc analyse, klassificering af logs, eller at generere træningsdata.

Idempotens og retries

LLM-kald fejler oftere end klassiske API-kald: rate limits, timeouts, indre fejl. Din pipeline skal kunne genfremføre.

⚠ Idempotens-nøgler

Send altid en idempotency_key (deterministisk hash af input + bruger + tidspunkt). Hvis et job genstartes, må det ikke producere et nyt resultat — og særlig ikke kalde et tool der koster penge to gange (fx send-email, charge-card).

Memory og state mellem requests

På B-niveau holdt du samtalehistorik i en in-memory liste. På A-niveau er det utilstrækkeligt: brugeren skifter device, dit serverpodd genstarter, eller du har et team af agenter der deler kontekst.

  • Short-term memory: Redis med TTL = samtalens varighed.
  • Long-term memory: Vector DB med embeddings af tidligere interaktioner — slå op i RAG-stil.
  • Episodic memory: Struktureret JSON i din relations-database. "Brugeren kan godt lide fodbold, foretrækker korte svar."
🔥 PRØV DET NU — Async-design
  1. Tegn et flow for: "Brugeren uploader 30 PDF'er, vil have en samlet sammenfatning."
  2. Hvor er kø-pointet? Hvor er status-opdateringerne? Hvad sker der ved fejl i én PDF?
  3. Hvor mange parallelle workers giver mening? (Husk rate limits.)

Det her er præcis det design en arkitekt ville stå med på et whiteboard på dag 1 af et A-niveau-projekt.

4
Self-hosted vs. API: det store strategiske valg
Total Cost of Ownership, lock-in og hvornår det vinder at hoste selv
+

Indtil nu har vi taget for givet at du bruger frontier-API'er (Claude, GPT, Gemini). På A-niveau kommer det strategiske spørgsmål: hvornår betaler det sig at hoste open-source-modeller selv?

Fordele ved API'er (de typiske 80%)

  • Bedste kvalitet — frontier-modeller leder still i 2026
  • Ingen GPU-drift, ingen modelopdateringer, ingen on-call
  • Pay-as-you-go — passer til variable workloads

Fordele ved self-hosting (de strategiske 20%)

  • Datasuverænitet: Visse data må ikke forlade din infrastruktur (jura, sundhed, statslige data).
  • Konstant load: Hvis du kører 24/7 høj volumen, kan dedicated GPU'er være billigere end pay-per-token.
  • Ingen lock-in: Hvis API-leverandøren slukker din konto eller ændrer priser, har du dit eget setup.
  • Custom fine-tunes: Din LoRA fra A1 kan kun deployes i dit eget setup (eller på platforme der supporterer det).

Break-even-beregningen

Skitse
API-pris pr. 1M tokens: ~5 USD (Sonnet-klasse model) GPU-leje, dedicated H100 24/7: ~2.500 USD/måned Throughput pr. H100: ~5M tokens/time → 3.6 mia/måned Break-even API: 500M tokens/måned ≈ 17M/dag Konklusion: - Under 17M tokens/dag → API vinder næsten altid - Over 50M tokens/dag → self-host kan være 5-10× billigere - Imellem: spørg jurister, security og operationelt
⚠ Den glemte omkostning: bemandning

Self-hosting kræver mindst én SRE/ML-engineer der ved hvad de laver. Lønomkostningen er ofte større end GPU-omkostningen. Glem det aldrig i TCO.

Hybrid: det realistiske valg

De fleste A-niveau virksomheder ender på en hybrid: API til de svære og sjældne queries, self-hosted (eller fine-tuned via API som Anthropic's eller OpenAI's hosted fine-tuning) til de hyppige standardiserede.

Værktøjer i 2026
  • vLLM: De facto standard til at serve open-source modeller — høj throughput
  • TGI (Hugging Face): Alternativ med god support til kvantiserede modeller
  • Modal, Replicate, Together.ai: "Self-hosted som service" — kompromis mellem API og fuld self-host
🔥 PRØV DET NU — TCO-øvelse
  1. Estimer eller find tal for et reelt AI-projekt: tokens/måned, gennemsnitlig request-størrelse, peak-load.
  2. Beregn månedlig API-pris ved Sonnet-niveau prisen.
  3. Beregn alternativet: H100 dedicated × hvor mange.
  4. Tilføj: 1 ekstra ingeniør (12.000 USD/måned i DK kontekst).
  5. Hvad er break-even? Hvad ville du anbefale?

Det her er præcis den øvelse en CTO eller VP Engineering bruger en uge på, hver gang AI-spend overstiger 50k DKK/måned.
